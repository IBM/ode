CHECKLIST

 

1.      Be sure you know

  • What is being fixed

Or

  • What feature is being added (special permission needed)?
  • The patch is not invalidating any public interfaces
  • The patch is not changing objects owned by other packages
  • The patch is not installing entirely new packages

 

2.      All files to be added are ready to ship

  • The binaries are compiled, optimized and stripped.
  • The source files are sccs delta'd.
  • If you are replacing a file, which is the source or target of a hard link, you are also replacing all files on that inode.

 

3.      You know the correct patch id number.

 

4.      You know what packages are to be included in the patch.

 

5.      You have selected the format appropriate to this patch.

 

6.      If directory format, you have constructed the README file and all necessary patch scripts such as prepatch or postpatch

 

7.      You know what patches (with different base numbers) are obsoleted by, required by or incompatible with this patch.

 

8.      You have constructed all non-standard class action scripts including old style install class action scripts if the patch may be applied to a 2.4 or earlier system. For patches delivered to a 2.5 or greater system the generic class action script needs to be modified to ensure that the backout package is created correctly.

 

9.      For each package, you need to make sure you have all of the required objects

a)      The proto area (either fully populated or sparse)

b)      A prototype file defining each of the objects to be replaced (no ftype "i" objects) buildpatch incorporates ftype "i" files automatically. Or if using this utility interactively, you can provide an object list indicating what objects in the proto area to be included and add symlinks and devices manually

c)      A deletes list if required. This file goes into the utils_directory, described below, along with the pkginfo and the prototype file.  The objects listed in the deletes file should contain paths that are relative to the BASEDIR variable defined in the pkginfo file.

d)      All necessary special scripts

e)      If a copyright file is used (for Sun specific patches it is required) place this in the a directory named "install" in the proto_area, described below, and buildpatch will incorporate into the patch package

 

MANUAL

NAME

 

buildpatch - a utility to construct a Solaris(TM) patch

 

SYNOPSIS

 

buildpatch [ -f directory -p patch_number ] [ -7 ] [ -c config_file ] [ -o "obsoletes_list" ] [ -r "requires_list" ] [ -i "incompatibles_list" ] [ -w ] [ -v ] [ -M patch_metafiles_directory ] [ -d target_dir ] [ -U proto_area utils_directory ] [ -P pkgname proto_area utility_directory ... ]

 

buildpatch [ -f directory -p patch_number ] [ -c config_file ] [ -o "obsoletes_list" ] [ -r "requires_list" ] [ -i "incompatibles_list" ] [ -w ] [ -v ] [ -M patch_metafiles_directory ] [ -d target_dir ] [ -O patch_level pkgname proto_area utility_directory ... ]

 

buildpatch [ -f directory -p patch_number ] [ -c config_file ] [ -o "obsoletes_list" ] [ -w ] [ -v ] [ -M patch_metafiles_directory ] [ -d target_dir ] [ -I patch_level pkgname proto_area utility_directory ... ]

 

buildpatch [ -f package -p patch_number ] [ -7 ] [ -c config_file] [ -o "obsoletes_list" ] [ -r "requires_list" ] [ -i "incompatibles_list" ] [ -w ] [ -v ] [ -S ] [ -d target_dir ] [ -U proto_area utils_directory ] [ -P pkgname proto_area utility_directory ... ]

 

AVAILABILITY

 

pkgadd the package SUNWpatch from benet.eng.

i.e. pkgadd -d /net/benet.eng/benet/Sustaining/PatchTools/SUNWpatch all

 

DESCRIPTION

 

This utility provides a mechanism for the purpose of modifying an installed software product. The "patch" repairs a design flaw in an installed software product or, under certain circumstances, installs a carefully controlled new feature. Since the unit Sun delivers as a software product is the System V package, all patches are performed on and with reference to a package or packages.

 

The old patch architecture (used prior to Solaris 2.5) required the patch builder to construct the patch by hand, building the individual sparse packages and then inserting them into a patch directory. In that architecture, the installpatch utility was responsible for all of the analysis and most of the actual work associated with installing the patch packages. The new architecture, supported by buildpatch, places most of the smarts into the individual patch packages. In support of this task, buildpatch constructs the patch packages so you don't have to.

 

Buildpatch needs to do this because it inserts special scripts into each package so that the package will ellicit correct behavior upon installation. Since buildpatch inserts a preinstall script into each package, if the patch requires some other preinstall script, it will need to be merged with the preinstall script buildpatch inserts. The same goes for any package script that may be special to this particular patch. The method for merging these scripts is shown below under Procedure Scripts. This takes advantage of the features provided in the System V package, specifically, making use of the update mechanism.

 

Once a package is installed onto a host it is possible to later repair design faults in that installation through a package update.  When a package is updated, a package of the same name, primary version and architecture is marked with some identifier like a patch ID and installed using the pkgadd utility. It really is that simple, but of course that doesn't mean we intend to leave it that simple.

 

Here's what makes it complicated. There are specific desirable features we have come to expect from a patch. In order to incorporate these features, this tool inserts package scripts, which provide the following capabilities:

 

1.      The ability to backout the patch

2.      The ability to establish contingencies based upon obsolescence or incompatibility

3.      The ability to automatically resolve dependencies (later)

4.      The ability to patch all applicable clients on a server (later)

 

While these scripts are easy enough to understand (they're explained in Appendix 2 of Sun Specification 950-2686-01, Solaris Patch Architecture) there are several of them; and, some care is required if the producer of the patch needs to implement a special feature in addition to those provided automatically by this utility.

 

Buildpatch version 1.6 and beyond supports the building of a Progressive Instance Patch. A PIP patch installs only to a 2.4 system and previous releases. It creates a patch pkg that adheres to the old style of patch installation. It builds a patch that will create a seperate instance of the sparse patch pkg once installed on the target host.

 

 OPTIONS

 

-7 

Builds a Solaris 7 or later direct instance patch. Installpatch, backoutpatch, and Install.info files are omitted.

-c config_file 

Designate an alternate configuration file (default is buildpatch.conf).

-f patch_format

Designate the patch format as patch_format (directory or package)

-p patch_number

Designate the patch id number

-o "obsoletes_list"

List of patches that this one obsoletes

-r "requires_list"

List of patches that this one requires

-i "incompatibles_list"

List of patches with which this one is incompatible

-w

During construction of this patch, write over any old patch of the same name in the build area

-S 

Provide package format patch in stream format (this is the default for any package format patch containing more than one package)

-M patch_metafiles_directory

Designate a directory containing the patch metafiles such as README, prepatch and/or postpatch. (Only applies to directory format patches)

-d target_dir  

Construct the patch in directory target_dir.  If no entry is provided, the default specified n buildpatch.conf as Patch_dir_dflt is used.

-P pkgname proto_area utility_directory

Use pkgname as the name of the package to be constructed, proto_area as the root of the proto area and use utility_directory as the path to the utility directory.

-U proto_area utils_directory

This is shorthand for the "-P" option above. It instructs buildpatch to construct a package for each utility directory found in utils_directory using the proto area provided. Each utility directory in utils_directory must be named according to the package to be built (eg: SUNWcar.m)

-O patch_level pkgname proto_area utility_directory

Same as "-P" except that the patch is constructed so as to allow installation to Solaris 2.4 and earlier systems as well as 2.5 and later. The patch_level parameter is the value to be assigned the "PATCH=" field in the old-style VERSION. (Only applies to directory format patches)

-I patch_level pkgname proto_area utility_directory

 Same as "-O" except that the patch is constructed so as to allow installation only to Solaris 2.4 and earlier systems. The patch_level parameter is the value to be assigned the "PATCH=" field in the old style VERSION. (Only applies to directory format patches)

 

EXAMPLES

 

The Simple Case

 

If the patch meets the following requirements, then buildpatch provides all special services.

 

1.      All files are being changed by replacement, not editting. This means there are no files of ftype "e" or "v" and all classes can be treated as though they were class "none".

2.      No installed files are being deleted by the patch, only replaced.

 

For a patch of this sort, only a workspace containing the modified deliverables, an object list (or prototype file) and the original pkginfo file are needed to fully define each package in the patch.  No class action scripts and no deletes file need to be provided. No commands text needs to be added to the package scripts found in the files directory and no special provisions need to be made for testing.  Most patches may be provided using the simple case.

 

A prototype file is required instead of an object list if

 

1.      An object is being delivered which is not represented in the proto area

2.      Files belonging to classes other than "none" are being delivered

3.      Package scripts not found in the install directory are being delivered

 

REMEMBER: The patch may not change the class of installed objects.

 

In the case of special classes, they don't require a special class action script but they do need to be named so that the correct removal scripts can be used if the package is later removed. It is also crucial that if new classes are introduced, the reference pkginfo file CLASSES entry must be updated appropriately. Any classes currently installed on the target host will be inherited from the installed package (although it never hurts to set the CLASSES parameter correctly to start with).

 

Definitions

 

We will begin with a definition of terms.

 

proto area 

This is a directory containing the binaries or source files to be delivered. It could be a fully populated workspace having only a few objects modified or it could be a sparse tree containing only the modified objects intended for delivery.

 

utility directory

A directory containing all required metafiles in support of the proto area.

 

install directory

A directory in the utility directory called "install". This is where the package-specific files are placed including depend, Copyright and special merged procedure scripts.

 

reference pkginfo

A pkginfo file which is at least the pkginfo file originally delivered with the installed package but which may include all required patch package entries.

 

prototype  

An optionally complete prototype file as defined in prototype(4). This prototype file contains entries representing all package objects to be delivered.

 

object list

The list of objects in the proto area which are intended to be delivered as part of the patch. If the proto area is sparse then no object list is required.  In interactive mode, the

object list will be converted to a package prototype file and presented to the user for editing since the object list does not contain enough information to guarantee completeness.

NOTE : it is not practical to provide absolute paths in the object list.

 

deletes file

The list of package objects to be deleted and not replaced. It is constructed in the same format as the object list except that absolute paths (beginning with "/") are handled correctly (like in the prototype file).

 

There are three ways to organize all of the required files for each package.

 

1.      explicitly - answer each of the  questions as they are asked with the pathname to the required object

 

2.      implicitly - provide the following two directories

  • The proto area
  • A utility directory which contains the following objects

-         pkginfo : reference pkginfo file

-         prototype: optional prototype file

-         objlist : optional object list

-         deletes : list of files to delete

-         install : a directory containing package-specific files and merged scripts

 

3.      implicit shorthand ("-U" option) - provide the following two directories

  • The proto area
  • A directory containing a utility directory (as above) for each package to be included which is named after the included package

 

Notes

 

The buildpatch utility is a highly configurable utility that will construct a patch compatible with the Sun patch architecture for Solaris 2.5. Take a look at buildpatch.conf for an idea of the level of configuration that is possible. It can be run interactively or from a command line. It is recomended that you run interactively for the first pass. Just type "buildpatch".

 

The command line may look something like this.

 

buildpatch -p 103344-02 -o "105467-02 107456-06" -i "107999-02" -f \

directory -w -P SUNWstuf trees/p3n4 utilitys/p3 -P SUNWjunk \

trees/p9n10 utilitys/p10

 

This command means the following:

 

Make a directory-format patch called 103344-02. Overwrite any existing patches of the same ID in the build area. It obsoletes patches 105467-02 and 107456-06. It is incompatible with patch number 107999-02. The patch consists of two sparse packages - SUNWstuf whose workspace is at trees/p3n4 and whose metafiles are found in utilitys/p3 (Yes, I know how to spell the plural of "utility") and SUNWjunk whose workspace is found in trees/p9n10 and whose metafiles are found in utilitys/p10.

 

Requirements

 

Certain parameters must be defined in each patch package pkginfo file. They may be supplied by the user or they may be inserted by this utility as desired.  (NOTE: Label substitution is provided for all scripts except class action scripts).  These parameters are all configurable and therefore only the general parameter is used here.  For example, if this is a patch from Bob's Software company and the parameter Patch_label is configured in buildpatch.conf as

 

Patch_label=BOBS_PATCHID

 

then the Bob's engineer could insert the following entry into the checkinstall script :

 

if [ $Patch_label = "11476-56" ]; then

 

The resulting entry after executing buildpatch on the package will be

 

if [ $BOBS_PATCHID = "11476-56" ]; then

 

Since these files would only be for Bob's use, this doesn't look like a very important function, but it is a collateral feature which some user may seek to exploit. In the interest of general application, this text will use the general-purpose label name instead of some company-centric term like SUNW_OBSOLETES.

 

The following parameter is required :

 

Patch_label

 

The following parameter is required under certain circumstances

 

Obsoletes_label   a space separated list of patches and/or supplements which are made obsolete by this patch

 

The following parameter is required under certain circumstances

 

Obsoletes_label   a space separated list of patches and/or supplements which are made obsolete by this patch

 

The following parameters are reserved and may not be used in any pkginfo file delivered as a patch.

 

PATCHLIST, PATCH_OBSOLETES

 

Directory Structure of this Utility

 

This utility is provided as a directory containing all required files. In the included "files" directory are the scripts that are used in the construction of each patch package. Specialized functionality may be merged into these scripts or new scripts may be written as shown below. These scripts must ellicit patch behavior as defined in Sun Spec 950-2686-01. Any special or merged scripts are inserted into the install directory for inclusion in the patch instead of the default scripts.

 

The minimal buildpatch looks like this:

 

buildpatch    buildpatch.conf    checklist    files

                                                                      /\

                                                       directory    package

 

The directory under the files directory called "directory" is for the scripts that will be used in construction of a directory format patch. The scripts in the "package" directory are used in construction of the package format patch. The directory to use is selected with the "-f" argument to buildpatch.

 

For a few weeks it was thought that the patch designer could provide script fragments that would be automatically merged into the default scripts. This may be permitted in some future release, but the restrictions required for such scripts would be complicated. For now, it's probably better to require the patch designer to look over the whole default script in the "files" directory and assess how specific modifications effect the overall functionality with his/her own eyes.

 

Procedure Scripts

 

All of the scripts provided in the "files/patch" directory which may be used by the old architecture contain a section that looks like this:

 

if [ "$PATCH_PROGRESSIVE" = "true" ]; then

# If this is being used in an old-style patch, insert

# the old-style script commands here.

 

#XXXOld_ScriptXXX#

 

exit 0

fi

 

Any special behavior required for a patch which may be installed to a Solaris 2.4 or older system should be manually inserted into this section. If a 2.4 patch requires a special preinstall script, copy the default preinstall script from the appropriate "files" directory into the "install" directory for the package requiring the script.  Put the special commands that would be part of the old-style preinstall script into the marked section of the copied preinstall script. By inserting the special script into the install directory, you inform buildpatch that this special preinstall script is to be used instead of the default.

 

When installing to a Solaris 2.5 or newer system, if additional operations are required beyond those normally provided, the recommended place for these package-specific commands is marked in this way:

 

# If additional operations are required for this package, place

# those package-specific commands here.

 

#XXXSpecial_CommandsXXX#

 

Based upon these markers, modify any script that needs modifying and place the special script into the appropriate install directory. As with the old-style section above, a "postinstall" script in the install directory overwrites the default postinstall script; an i.none script in the install directory overwrites the default install class action script for the "none" class. Only those scripts that need to be different from the default scripts need be stored in the install directory. As a company develops some experience with this method, a central repository could be constructed of standard variations on these scripts for automatic inclusion as the need arises.

 

Class Action Scripts

 

Since class action scripts are a little more complicated, up to three different versions may be required in order to correctly patch an installed package. If the package is just delivering regular files, the default class action script (i.script) will do the whole job, but if the file has to be installed in a special way, you will need to write these scripts.

 

The number of files is unfortunate, but the scripts themselves don't need to be very complicated. Here's what the scripts may look like for the class "drvlst".

 

i.drvlst

 

This is the install class action script required to add a line to an editable file which may or may not exist at patch time (perhaps it may have been created by a prior patch).

 

EXAMPLE:

# i.drvlst

 

pn=$$

 

if [ -n "$PATCH_BUILD_DIR" -a -d "$PATCH_BUILD_DIR" ]; then

    BUILD_DIR="$PATCH_BUILD_DIR/$PKGINST.$PATCH_PID"

else

    BUILD_DIR="$PKG_INSTALL_ROOT/var/tmp/$PKGINST.$PATCH_PID"

fi

 

FILE_DIR=$BUILD_DIR/files

RELOC_DIR=$FILE_DIR/reloc

ROOT_DIR=$FILE_DIR/root

 

BD=${BASEDIR:-/}

 

cd $BD

 

while read src dst; do

    #

    # If we have to undo the edits to this file later, store

    # the source line we inserted for the undo script.

    #

        if [ "${PATCH_NO_UNDO}" != "true" ]; then

            if [ "$BD" = "/" ]; then

                FS_Path=`echo $dst | sed s@"$BD"@@`

            else

                FS_Path=`echo $dst | sed s@"$BD/"@@`

            fi

 

        Build_Path="$RELOC_DIR/$FS_Path"

 

        cd $FILE_DIR

 

    #

    # Construct the prototype file entry. We don't care

    # if we are actually going to add a line or not

    # because the undo script is idempotent. (It isn't

    # always this easy)

    #

    echo "e drvlst $FS_Path=$Build_Path 0644 bin bin" \

  >> $BUILD_DIR/prototype

 

    # Now copy over the file

    DirName=`dirname $Build_Path`

    mkdir -p $DirName

    cp -p $src $Build_Path

    cd $BD

  fi

 

  if [ -f $dst ]; then

    #

    # If the file is already there, then add this line

    # only if it isn't already there

    #

    line=`cat $src`

    nawk -v test=$line '

BEGIN { found = 0 }

{

  if ($0 == test) { found = 1 }

  { print }

}

END {

if (found == 0) { print test }

} ' <$dst > $dst.$pn

    mv $dst.$pn $dst

  else    # otherwise, initialize it to this line

    cat $src > $dst

  fi

done

 

 

exit 0

 

u.drvlst (required if i.drvlst edits files rather than replacing them)

 

This is also an install class action script; but, this is responsible for restoring the editable file to the state it was in prior to installing the patch. It will recieve a source/destination pair like any install class action script and the source will be the original file that i.drvlst modified and the destination will be the patched file. In short, source and destination functions are reversed for u.drvlst with respect to i.drvlst

 

 

EXAMPLE:

# u.drvlst

 

pn=$$

 

while read src dst; do

  # If the file is there, delete this line

  if [ -f $dst ]; then

    line=`cat $src`

    nawk -v test=$line '

{

  if ($0 == test) { next }

  { print }

} ' <$dst > $dst.$pn

    mv $dst.$pn $dst

  fi

done

 

exit 0

 

r.drvlst (optional - not recommended)

 

This is a remove class action script intended to replace the installed remove class action script. If the patch is backed out, this script will not be restored to its original form! This means that if it is necessary to replace this script, the replacement MUST be backward compatible with the original.

 

Backout Scripts

 

Various scripts may be provided for the purpose of backing out the patch. The default scripts are all identified in the "files" directory by the "patch_" prefix.  These scripts combined with any "u." class action scripts are used to construct the backout package.  It is not recommended that the "patch_" scripts be modified by the patch builder since the person who wrote them barely understands how they work.

 

Utility Configuration

 

The buildpatch utility can be configured to operate in specific ways and to default to specific values using a configuration file in the directory containing the buildpatch utility. Set the values in buildpatch.conf to make sure the configuration agrees with your intended configuration. Here are the meanings of the configuration parameters.

 

·        Checklist=["yes", "no", "ask"] (default: "ask")

"yes"   - display this checklist whenever buildpatch is called

"no"    - do not display the checklist

"ask"   - offer the checklist

 

·        Patch_label=["string_value"] (default: "PATCHID")

 "string_value"  - string to use for the patch number (eg: SUNW_PATCHID)

 

·        Ask_obsoletes=["yes", "no"] (default: "yes")

"yes"   - ask for a list of obsoleted patches

"no"    - assume only implied obsoletion

 

·        Obsoletes_label=["string_value"] (default: "OBSOLETES")

"string_value"  - string to use as parameter representing the list of patches which this obsoletes (eg: "SUNW_OBSOLETES")

 

·        Ask_requires=["yes", "no"] (default: "yes")

"yes"   - ask for a list of required patches

"no"    - assume no required patches

 

·        Requires_label=["string_value"] (default: "REQUIRES")

"string_value"  - string to use as parameter representing the list of patches which this requires (eg: "SUNW_REQUIRES")

 

·        Ask_incompat=["yes", "no"] (default: "yes")

 "yes"   - ask for a list of incompatible patches

 "no"    - assume no incompatible patches

 

·        Incompat_label=["string_value"] (default: "INCOMPAT")

"string_value"  - string to use as parameter representing the list of patches with which this is incompatible (eg: "SUNW_INCOMPAT")

 

·        Ask_fmt=["yes", "no"] (default: "yes")

"yes"   - ask for the patch format

"no"    - use the default patch format

 

·        Default_fmt=["directory", "package"] (default: "directory")

"directory"- containing one or more packages

"package" - patching only one package

 

·        Ask_patch_dir=["yes", "no"] (default: "yes")

"yes"   - ask for the path to the patch construction directory

"no"    - use the default construction directory

 

·        Patch_dir_dflt=["string_value"] (default: "/var/spool/patch")

"string_value"  - path to the directory in which patches will be constructed

 

·        Patch_Overwrite=["yes", "no", "ask"] (default: "no")

 "yes"   - if this patch is already present, overwrite it

 "no"    - if this patch is already present, quit

 "ask"   - if this patch is already present, ask

 

·        Patchvers_pfx=["string_value"] (default: "-")

"string_value"  - the ASCII string separating the patch base number from the version (eg: "-" in 101345-02; the second version of patch base code 101345)

 

·        Patchstrict_str=["string_value"] (default: "R")

"string_value"  - the ASCII string which indicates patch number to be a restricted patch. (eg: the "R" in R101345-02; the trial release of patch 101345-02)

 

·        PATCHNO_FMT=["-r string_value [-r string_value...]"]

(default: "-r ^[0-9][0-9][0-9][0-9][0-9][0-9]$Patchvers_pfx\

 [0-9][0-9]$ -r ^$Patchtemp_str[0-9][0-9][0-9][0-9][0-9]\

 [0-9]$Patchvers_pfx[0-9][0-9]$")

"string_value"  - a  sequence of regular expressions representing the acceptable formats for the patch number. The parameters Patchvers_pfx and Patchtemp_str may be included in the expression. Each regular expression will begin with the string "-r ".

 

·        PATCH_UTILITIES_PATH=["path"] (default: "PatchTools")

"path"  - this is the path to the directory containing installpatch and backoutpatch. If relative, the directory containing the buildpatch utility is used as the starting point.

 

·        PATCH_META_PATH=["path"] (default: "none")

"path"  - this is the path to the directory containing patch-specific metafiles such as prepatch and README. If relative, the current directory is used as the starting point. This may include the arameter $PatchNum to allow automatic selection based upon patch number.

DIAGNOSTICS

 

ERROR : Cannot create directory <directory_name>.

Explanation:

For some reason, buildpatch cannot create the indicated directory. Remember the restrictions NFS places on a root user creating filesystem objects in a home directory. That is often the cause of this message.

 

ERROR: Cannot write to <filename>.

Explanation:

Buildpatch did not have permission to write to the target file.  See explanation above.

 

ERROR: Cannot find <filename>.

Explanation:

The indicated file was not found in the designated directory.

 

ERROR: Conflict between prototype file and deletes list - they install <filename> and then delete the same file.

Explanation:

An identical entry, <filename>, was found in the prototype file and in the deletes list. This indicates an error in the construction of one of these files since anything which exists in both will not be laid down on the filesystem.

 

ERROR: <patchnum> is an invalid patch number.

Explanation:

The patch number, <patchnum>, does not comply with the regular expression syntax specified by default or in the configuration file. Check the regular expressions used to define PATCHNO_FMT.

 

ERROR : The sed, awk and build class are not permitted in this version of buildpatch because the undo process is not yet figured out.

Explanation:

At this time, no sed, awk or build classes are permitted in a patch.

 

ERROR : All editable files must be installed, removed and undone using specialized class action scripts. No script has been detected for editable files in the <classname> class.

Explanation:

There are editable files (ftype e) of class <classname> in the patch for which no associated class action script has been provided in the install directory.

 

ERROR : An editable file is included in the class which contains ordinary files.

Explanation:

Some class has been provided which includes both editable and ordinary files. The ordinary files need to be separated into the none class or a class with a different class action script.

 

ERROR: patch number tokens are not all specified.

Explanation:

Both Patchvers_pfx and Patchstrict_str must be specified as non-null values

 

ERROR: No packages have been provided to patch.

Explanation:

At least one package must be provided to the buildpatch utility.

 

ERROR: A progressive instance patch must include patch levels.

Explanation:

The "-O" argument was formatted like a "-P" argument.

 

ERROR: Only directory format patches support progressiveinstance patching.

Explanation:

A "package" format patch was requested with a "-O" argument.

 

ERROR: Cannot construct object list from the contents of<proto_area>.

Explanation:

The buildpatch utility has found an entry in the pkginfo file which incorrectly defines a parameter which is usually constructed by buildpatch (eg: OBSOLETES). Remember that buildpatch constructs all patch-unique entries for the pkginfo files.

NOTES

 

Since the patch number format is established using a regular expression which is read directly into the environment. Appropriate escaping is necessary in order to prevent buildpatch from matching entries in the local directory that match the expression. The problem would if buildpatch.conf contained the entry:

 

PATCHNO_FMT="-r [0-9][0-9][0-9]-[0-9]"

 

and there were directory entries in the buildpatch directory that look like this:

 

102-2

103-4

 

As buildpatch read in the environment, the shell would evaluate the regular expression and hand buildpatch the following:

 

PATCHNO_FMT="-r 102-2 103-4"

 

For this reason, the regular expression must be bounded on both ends correctly.  This inhibits interpretation by the shell during the initial read. The correct version of the above assignment is:

 

PATCHNO_FMT="-r ^[0-9][0-9][0-9]-[0-9]$"
