    About this Document

    This document is intended to give the information necessary for the reader to use the ODE packaging commands and packaging configuration files.

    See the ODE Documentation Index for a complete list of ODE documents and their purposes. It is recommended that the ODE Concepts and Common Makefiles documents be read and understood before continuing with this document.

    Packaging Overview

    ODE supports the creation of platform-specific installable images as part of a build within a development environment. Typically, each platform has its own set of package-creation tools that require a particular set of input files to generate the package. ODE handles the creation of these input files, the collection of all installable files in one location, and the execution of the platform's packaging tools, to generate a product package.  It is possible to use ODE to create packages for any of the supported installation tools from any platform, in that the default packaging tool used for any platform can be redefined by the user.  This is beneficial when creating packages for multiple installation tools from the same platform.

    Common Metadata File (CMF)

    The Common Metadata File, or CMF, is a platform-independent text file that contains the necessary information needed to specify the contents of a product package. This file is parsed by the ODE packaging tools, which will in turn generate the platform-specific set of packaging control files used by each platform's packaging tools. The CMF supports a series of InstallEntity and File stanzas, which allows complex product hierarchies to be represented in the final package. See the Commom Metadata File section for more detail on the specifications of the CMF.

    Types of Packaging

    ODE supports two main types of packaging; initial product packaging (IPP) and service packaging (SP). IPP is a full and complete package, containing the entire contents of a product. SP is generally a partial, or incremental set of files used to update or refresh an existing software installation. The most important difference between initial product packaging and service packaging is that not all parts are necessarily included in a service package; therefore, the determination and collection of the shippable parts is required for service packaging. These features are all supported by ODE.

    ODE Packaging Targets

    This section provides details about the standard ODE packaging targets. Each of these targets are run by issuing mk <target name> command in a sandbox. Targets like gather_all, package_all, buildandpackage_all execute other targets. The install_all/instdep_all and parse_all targets handle the collection of all installable files in one location and the creation of control files, respectively. The runpkgtool_all target executes platform specific packaging tool to create an installable image.

    All the packaging targets use the following set of common makefile variables:
     
    Makefile Variable Description Required/Optional
    JVM The Java virtual machine to use. Default value is java. There some cases where the virtual machine would either need to be different or fully qualified, due to conflicting JDKs. Required
    PKG_JAVA_FLAGS Use this variable to specify any additional flags to the java command. For example, to increase the maximum heap size used during the execution of the tool, specify PKG_JAVA_FLAGS=-mx32m. Optional
    PKG_JAVA_DEFS Similar to PKG_JAVA_FLAGS, but intended for "-D" type flags that should be passed to all packaging invocations of Java. Optional
    PKGFAMILY The packaging software to use.  The default is "native", which implies the built-in packaging tool for the platform on which the build is occurring.  Valid values are "mkinstall", "swpackage", "pkgmk", "rpm", "ispe", "ismp", and "mvs". Required
    PKGVERSION The version of the packaging software. This is not yet used; it will be checked when support for multiple versions of the same packaging tool (which behave differently) is added. Optional
    OMIT_PACKAGING_RULES If this variable is defined, mk will not include any of the packaging rules files.  For users who do not perform any packaging, this can reduce the amount of time it takes for mk to parse all of the rules files. Optional
    PKG_CLASS The class of package being created, can be IPP (Initial Product Package) or SP (Service Package).  For standalone ODE packaging, PKG_CLASS must be IPP. The default value is IPP. Required

    The following sections describe each of these targets in detail.

    buildandpackage_all

    The buildandpackage_all target can be used to combine the actions of build_all and package_all targets.

    package_all

     The package_all target is used to combine the actions of the gather_all, parse_all and runpkgtool_all targets. The order in which these targets are executed depends on the platform and type of packaging. The following table describes the behavior:
     
    Platform Type of Packaging Order
    All Platforms except MVS/USS IPP gather_all, parse_all, runpkgtool_all
    MVS/USS IPP parse_all, gather_all, runpkgtool_all
    All Platforms SP gather_all, parse_all, runpkgtool_all

    gather_all

    In IPP, this target executes install_all target on all platforms and additionally it executes installmvs_all on MVS/USS. In SP, this target executes instdep_all target on all platforms.

    install_all

    Once all the files to be packaged have been built, the next step is to gather all of the files to be shipped in one location. This is accomplished by executing install_all target which copies all files specified by ILIST to the directory ${TOSTAGE}/${IDIR}. This target is executed by issuing mk install_all in a sandbox.

    The makefile variables used by the install_all target are documented in ODE Common Makefiles.

    installmvs_all

    This target handles the copying of shippable files from the MVS/USS side of an MVS/OE system to a location on the TSO side, where they can be subsequently packaged.  Additionally, parts can be copied from partitioned datasets on TSO to the distribution library used for final packaging.  This target is valid only on MVS/USS platform and it is executed by issuing mk installmvs_all command in a sandbox.

    instdep_all

    This target is meant to be used in service packaging. This target when used in a sandbox that is backed against a backing build sandbox, will copy changed shippable files to the ship tree. It determines the changed shippable files by comparing files in the sandbox with the corresponding files in backing build sandbox. This target is executed by issuing mk instdep_all command in a sandbox.

    Common Metadata File

    This section describes the general structure of the Common Metadata File (CMF) and defines the specific attributes necessary for creating this file.

    The CMF is the input mechanism for the parse_all pass. It contains the data necessary for the parser to create the files needed as input to the packaging tools. The minimum requirement in a CMF is one InstallEntity and one File stanza. However, several platforms require a more complex structure than one InstallEntity and one File stanza, and will therefore not accept such a configuration. See the below section Inheritance within Platforms for more information on the typical structure of packages on various operating systems. The actual contents in the CMF will vary depending upon the platform.

    Stanzas are a collection of attributes that can contain either other stanzas or attributes.  The CMF consists of two types of top-level stanzas.  An InstallEntity (IE) stanza is used to manage other installable entities and groupings of files, while a file stanza contains the attributes of a specific installable file, or in some cases, the stanzas that contain the specific attributes of the installable file.  All other stanzas in the CMF are sub-stanzas of one of these two top-level stanzas.

    The CMF can be automatically generated from the existing set of makefile variables. Please check CMF Generation - Users Guide for more details.

    CMF Rules and Syntax

    Comments

    The // character denotes a single-line CMF comment. Any text to the right of this character is considered a comment. A block CMF comment should be enclosed within /* and */. Comments, singular or block, will not be interpreted by the parser.

      // this line is a comment.

      /*
      this is a
      multi-line comment
      */

    Attribute Block Characters

    The { and } characters are markers denoting the end of a stanza's attribute block. These also represent the beginning and the end of a sub-stanza with a stanza block. These act as delimiters for metadata specification stanzas.

        <stanza name>
        {
               <sub-stanza name>
              {
                  <attributes>
               }
         }

    Delimiter

    The semicolon (;) is a delimiter terminating an attribute name-value pair.  The following example also contains a comment.

        installEntity
        {
            name = someEntityName; // note the semicolon attribute delimiter
        }

    Labels

    You can also provide labels in the CMF as required. A label can be any word that does not correspond to a reserved word for the CMF. A reserved word is any attribute name, stanza name, or entity name. Use of a label is simply another way to identify and denote CMF entries, similar to a comment.  A label can appear anywhere in the CMF, but must be followed by a colon (:).

    A label specified in this manner in the CMF is not reflected in the generated control files.

       label1   :  installEntity {
       label2   :  name = someEntityName; // note the semicolon attribute delimiter
       label3   :  }

    CMF Structure Rules

    The following rules apply to a CMF:
    • The entityName attribute of an InstallEntity stanza must be unique in the CMF.
    • The combination of the sourceDir and sourceFile attributes (sourceDir + sourceFile) in a File stanza must be unique in a CMF.
    • Only one parent may be specified for any InstallEntity. The InstallEntity can have only one of the following:
      • One or more child entities as specified in the immChildEntities attribute.
      • One ore more child files as specified in the immChildFiles attribute.
      • These two cases are mutually exclusive and therefore one and only one must be specified for a given InstallEntity.
    • The stanzas in the CMF can listed be in any order.
    • Every InstallEntity stanza must have a LinkInfo and an EntityInfo stanza.
    • Within the stanza (or sub-stanza), the attributes can be in any order; however, all the sub-stanzas must be within a parent stanza.
    • An InstallEntity at the top of the tree (has no parent) must have its parent attribute specified as "NULL".

    Attribute Value Data Types

    This section describes the different data types that can be used to represent values in the CMF.
    String
    A group of characters or words, specified within double-quotation marks. To specify quotation marks within the string, use the escape character, a backslash. To specify a backslash in the string use \\.
    Example: attribute ="This is a string";
    Constant
    A single word, number, or combination of words or numbers, without spaces, specified within single-quotation marks.
    Example: attribute = 'aConstantName';
    Filename
    Path to a file, specified as "</path/fileName>". The filename itself cannot contain a ">" symbol.  Either a full or absolute path can be specified.
    Example: attribute = </usr/bin/fileName>;
    ReqType
    A space-separated series of string values, surrounded by parentheses, consisting of one to three values.
    Example: attribute = ("fileName1" "fileName2" "optionalfileName3");
    List of entities
    Any of the entities described previously (String, Constant, Filename, ReqType) can be represented in a list format. To list entities, use square braces to enclose the list and use white space between the entries in the list. The following example shows a list of Filenames:
    attribute1= [

                         <c:\path1\file1>
                         <c:\path1\file2>
                         <c:\path2\file3>
                         <c:\pathn\filen>
                       ]
    Where:
    attribute1is the name of the attribute that takes a list of files as a parameter (example: inputPath, shipPath, etc.)
    [       is the beginning of the list,
    < >   encloses each filename, and
    ]       closes the list .
    There is no limit to the number of entries in a list, as long as all of them remain enclosed within the brackets.

    CMF Stanza Specifications

    File Stanza

    The following is a list of attributes that are found in the CMF for the File stanza along with a definition of each attribute. The platform-specific use of the attribute is described along with its corresponding entry in the packaging tool's control file in Additional Packaging Information. The data type of each attribute appears in parentheses after the description.
    distLib
    Alternative name for targetDir. See targetDir for a description.(string)
    entry
    The JCLIN entry point for the part. (string)
    extAttr
    The external attributes for this part used by the JCLIN. (string)
    fileDirectives
    Specify a list of directives for each file.
    doc - is used to mark a file as documentation. RPM keeps track of documentation files in its database so that a user can easily find information about an installed package.  This is on a per-file basis.
    fileDirectives = [ "doc" ] ;
    docdir - a directory related directive used to add a directory to the list of direcotries that will contain documentation.
    fileDirectives = [ "docdir" ] ;
    dir - used to mark a directory.  This is redundant since directories are marked already using fileType it will appear twice on a line, but rpm will not complain.
    fileDirectives = [ "dir" ] ;
    attr - the attr directive is a combination of (mode, user, group).  If you set the attr directive as a fileDirective, ODE will not complain, but will ignore the entry.  This is because ODE gets these values from the mode, user, and group attributes set in the cmf for each file stanza.
    config - is used to mark the specified file as a configuration file.  RPM performs special processing on config files when packages are erased, and during installations and upgrades.
    fileDirectives = [ "config" ] ;
    verify - is used to control which of nine different file attributes are to be verified by RPM.
    The attributes are:
    • owner -- The file's owner.
    • group -- The file's group.
    • mode -- The file's mode.
    • md5 -- The file's MD5 checksum.
    • size -- The file's size.
    • maj -- The file's major number.
    • min -- The file's minor number.
    • symlink -- The file's symbolic link string.
    • mtime -- The file's modification time.
    fileDirectives = [ "verify(mode md5 size maj min symlink mtime)" ] ;

    Specifying Multiple Installable Files within a File Stanza

    The File stanza can be extended to specify that a single built file can ship as multiple installable files.  This functionality requires the use of the PackageData and SourceData stanzas.  These stanzas are optional sub-stanzas of the File stanza, and are used in place of direct attribute assignment in the stanza.  The SourceData stanza contains the common information for the installable file, namely the sourceDir and sourceFile attributes.  A File stanza can only contain one SourceData stanza.  The PackageData stanza contains the remaining attributes required to specify a File stanza, as listed above.  Multiple PackageData stanzas can exist within a given File stanza, provided that no two PackageData stanzas refer to the same installable file, as defined by the combination of targetDir or distLib and targetFile.  See File Stanza Example 3 for an example of the use of the SourceData and PackageData stanzas.
     
    Note: Currently, this functionality is only applicable on USS.  The CMF syntax is legal on all platforms, but the generation of the packaging control files on all non-USS platforms will only use the first listed PackageData stanza in the CMF, maintaining the one-to-one relationship between built files and installed files.

    Specifying Package Requisites in ODE

    Specifying Requisite Information

    Packaging requisites are used to specify dependencies for the package being created. The dependencies are specified while the package is built by specifying them in the CMF and the platform specific installation tool will enforce these dependencies during the installation and uninstallation of the package.
    Dependencies for the current entity must be specified using the requisites attribute of the RequisitesInfo stanza. The format to use for specifying this information is as follows:

        requisites = [ (reqType, packageInfo, description) ];

    where reqType is the type of requisite, and should be one of the supported values described in the table below:

    packageInfo is where the requisite package information is defined, and should be specified as follows:
        AIX: the requisite fileset name and all corresponding version release modification fix (VRMF) information
        HP-UX: the requisite fileset name
        Solaris/Unixware: the package name
        Linux: the package name
        MVS/USS: the FMID name

    and description is an optional string to describe the requisite.

    Requisite Descriptions

    The following sections describes the different types of requisites available to packages on different platforms.
    ( P | pre )
    pre-requisite -
    AIX Generates keyword prereq in fileset.prereq file. A prereq indicates both a functional and an installation order dependency of one fileset on another fileset. Pre-requisites should be used as little as possible in packaging.
    HP-UX Generates keyword prerequisite in pcd.psf file. A prerequisite dependency requires another software object to be installed (configured) in order for the first fileset to be installed.
    MVS In SMP/E, a SYSMOD that must be installed before or along with another SYSMOD in order for that other SYSMOD to be successfully installed. It is defined by the PRE operand on the ++VER statement.
    RPM Generates Prereq: tag in pdc.spec file.
    Solaris Generates keyword P in the depend file. This indicates a package that depends on the existence of another package.
    ( C | co )
    co-requisite -
    AIX
    Generates keyword coreq in the fileset.prereq file. A coreq indicates only a one-way functional dependency of one fileset on another fileset. Co-requisites are the most desirable requisite to use in the case of a functional dependency since a coreq only indicates that both filesets must be installed on the system; no order to the install is implied. A common misunderstanding with coreqs is that they are not two-way as the name may imply.
    HP-UX
    Generates keyword corequisites in the pcd.psf file.
    1) A corequisite dependency states that a fileset requires another software to operate correctly.
    2) A corequisite dependency requires another software object to be installed in order for the dependent fileset to be usable.
    MVS
    Corequisite SYSMODs, each of which can be installed properly only if the other is present. Corequisites are defined by the REQ operand on the ++VER statement.
    RPM
    Generates Requires: tag in the pcd.spec file.This tag is used to warn RPM that the package needs to have certain capabilities to operate properly. These capabilities refer to the name of a package or to a virtual package provided by one or more packages that use the Provides tag. A package that requires certain "capabilities''. Manual dependency requirements are always represented in terms of packages.
    ( L | ins )
    install requisite -
    AIX
    Generates keyword instreq in the fileset.prereq file. An instreq is used in cases where one fileset should be installed only IF another fileset is installed.  For example, it doesn't make sense to install the documentation for a package unless the "core functionality" is also installed.  But the documentation will not prereq the "core functionality" it will instreq it.
    ( R | rev )
    reverse requisite -
    Solaris
    Generates keyword R in the depend file. It indicates a condition when another package depends on the existence of your package. Use the reverse dependency type only when a package that cannot deliver a depend file relies on your package.
    ( V | pro )
    provides -
    RPM
    Generates Provides: tag in the pcd.spec file.This tag is used to specify a virtual package that the packaged software makes available when it is installed.  Normally this package would be used when different packages provide equivalent services..
    ( I | if )
    if-requisite -
    AIX
    Generates keyword ifreq in the fileset.prereq file. An ifreq is probably the most complicated form of requisite and is used primarily with updates. An ifreq is best thought of as a conditional coreq. The ifreq is similar in behavior to a coreq, in that, if one update ifreqs another update then both updates must be installed at the same time or the target update must already be installed, with the order of installation not being important. The conditional part is that the second fileset must only be installed at the specified level if it is already installed on the system at some lower base or specified level. Otherwise, the ifreq is ignored until the second fileset is installed on the system, at which time lppchk can be used by the customer to tell them if there are any missing updates. The ifreq does NOT allow installp to add the required fileset to the list of filesets to install (or update) even if the -g flag is used.
    MVS (IFREQ)
    A conditional requisite. Conditional requisites are specified on the REQ operand of the ++IF statement.
    ( X | neg )
    negative pre-requisite -
    HP-UX (exrequisites) (Not yet implemented.)
    A list of dependencies on software that may not be installed when this software is installed. See also the ancestor, corequisites, and prerequisites attributes.
    MVS
    In SMP/E, a function that is mutually exclusive with another function. It is defined by the NPRE operand on the ++VER statement.
    RPM
    Generates the Conflicts: tag in the pcd.spec file. Implies that this package conflicts with the package being installed.
    Solaris (incompatible)
    Generates keyword I in the depend file. This implies that the existence of the indicated package or version is incompatible.
    ( S | sup )
    supersedes -
    AIX
    This entry gets generated in the fileset.supersede file. The fileset.supersede file is optional and used for both install images and updates. When used with an install image, the fileset.supersede should list the one fileset that this image completely replaces.  For updates, the fileset.supersede file contains a list of the fileset and levels (with fix field smaller than this one) that are superseded by this fileset update.
    HP-UX
    Generates ancestor tag in the pcd.psf file. An "attribute" which signifies the name of a previous version fileset used in the "Match-What-Target-Has" functionality. If the match_target option is set to true, the system will match the ancestor fileset name to the new fileset name.
    MVS
    Superseding SYSMOD. In SMP/E, a SYSMOD that contains all the functions in another SYSMOD and is recognized as the equivalent of that other SYSMOD. The superseding SYSMOD uses SUP operand on its ++VER statement to specify the superseded SYSMOD.
    Solaris
    This entry gets generated as Version in the file compver. It is a method of specifying package backward-compatibility.
    ( D | del )
    delete -
    MVS
    SMP/E deletes a load module and any known alias names from a target library and the associated target zone. The ++DELETE statment uses the DEL operand to specify the module that is to be deleted.

             Note: On MVS/USS, the delete requisite format for the ++DELETE statement will be as follows:
              requisites = [ ("D", filename, syslib/alias) ];
                     where filenameis the name of the load module to be deleted
                                     syslib/aliasis optional, default = SYSLIB(ALL)

    Configuration Files in an ODE Package

    Specifying Configuration Files

    The configFiles attribute is a member of the PathInfo Stanza of an InstallEntity. It is used to create an entry for package installation control scripts, also known as "configuration files". This document uses specific examples for packaging on AIX, RPM, Solaris and HP-UX.

    The configFiles attribute, which is a list of ("keyword" "value") pairs, belongs to the PathInfo Stanza.  The keyword indicates which control script and the value indicates the filename of the control script.  The PathInfo stanza has the following syntax: 

      PathInfo 
        {              
                configFiles = [ (keyword_1  filename_1)    
                                (keyword_2  filename_2)  
                                        .
                                        .
                                        .
                                (keyword_n  filename_n) ] ;
        }

    Example CMF Entry

      PathInfo 
        {       
                configFiles = [ ("postinstall"  "post_install.sh")    
                                ("preremove"    "pre_remove.sh") ]                    
        }
    where "postinstall" and "preremove" are descriptive keywords , and "post_install.sh" and "pre_remove.sh " are the corresponding installation control scripts.

    Platform specific issues

    Keyword specification
    HP-SD, RPM and Solaris pkgmk commands require both a valid keyword and file name. Each has its own set of keywords. The package tool runs the script associated with the appropriate keyword. The filename need not be the same as the keyword. AIX does not use keywords so it does not require a valid keyword be used. Instead AIX has a specific convention for naming the control scripts. During installation, update, removal, commit and rejection of a package AIX runs scripts which look for the existence of these files and runs them if they are found. The keyword must be one of the keywords listed in the table below. Only a valid filename is needed, and any keyword will be ignored. However, a keyword must be present, even if its an empty string ( "" "pre-install.sh" ), to satisfy the syntactical requirements of the CMF.

    Other Notes

    Specifying the Location:

    On Solaris, HP-UX, RPM, and AIX, the fully-qualified filename, or a relative filename, may be specified for the script.  If a relative path to a filename is specified, the file is expected to reside in the directory specified by the TOSTAGE variable.

    On Solaris, these scripts will then be copied from the directory specified in the configFiles attribute to the directory location specified by the variable PKG_CONTROL_DIR when the CMF is parsed by ODE by the parse_all target.  However, on AIX, HP-UX, and RPM the scripts must be manually copied to the directory specified by TOSTAGE prior to creating the package with the ODE runpkgtool_all target.  A convenient method to accomplish this is to specify the scripts as targets for the install_all target, by specifying ILIST and IDIR in the associated makefile.  This will ensure the scripts get copied to the TOSTAGE directory prior to the creation of the package.

    Naming the scripts:

    For AIX packaging, there is a restriction on the actual base names of the configuration files.  On Solaris, HP-UX, and RPM there is no restriction on the filenames as long as a valid keyword is used.  On AIX the file names must be of the format

    <fileset_name>.<keyword>_<install_type>
    Where fileset_name is the name of the fileset for the configuration files, keyword is an action descriptor for the file, for instance pre or post, and install_type is a descriptor for when the script is to execute, i.e. 'i' for installation, 'd' for deinstallation. An example script name on AIX is odehello.bin.pre_i. See the AIX Packaging Owner's Guide to LPP Packaging for more detail on the available keywords and install types.  The supported filenames are listed in the Supported Control Script Filenames and Keywords

    ROOT content configuration files
    On AIX, when specifiying control files for both USR and ROOT portions of a fileset, the configFiles attribute is used only for the USR portion control files.  An additional parameter, rootControlFiles, is used to list control files that should belong to the ROOT portion of the fileset.

    Examples

    Solaris
    On Solaris, scripts residing in the /home/mybb/src/scripts directory of a build environment would be specified with the below configFiles entry in the CMF.

             configFiles = [ ("postinstall"  "/home/mybb/src/scripts/post_install.sh")
                             ("preremove"    "/home/mybb/src/scripts/pre_remove.sh") ]

    When the mk parse_all command is executed, the scripts are automatically copied from the /home/mybb/src/scripts directory to the directory specified by the PKG_CONTROL_DIR variable, and the following enties are created in the packaging control file, pcd.proto:

             i postinstall=/home/mybb/src/scripts/post_install.sh

             i preremove=/home/mybb/src/scripts/pre_remove.sh


    AIX, RPM, and HP-UX
    On AIX, Linux, and HP-UX scripts must be manually copied to the directory specified by the TOSTAGE variable.  This can be accomplished by the gather_all target.  For example:

    If you keep your AIX control scripts in /home/mybb/src/scripts  and you want to make an entry for a pre_deinstall script for the ode.bin fileset, add the following entry to /home/mybb/src/scripts/makefile. Then when you run the 'mk install_all' command your scripts will be copied to the TOSTAGE directory.

    IDIR = /
    ILIST = odehello.bin.post_i odehello.bin.pre_d
    Since the directory location of the scripts is assumed to be TOSTAGE, the full path of the scripts is not needed in the configFiles attribute.  Therefore, the corresponding CMF entry for HP-UX and AIX would be as follows:

         configFiles = [ ("postinstall"  "odehello.bin.post_i")
                         ("preremove"    "odehello.bin.pre_d") ]

    When the mk parse_all command is executed on HP-UX, the configuration scripts are assumed to reside in the TOSTAGE directory, so the following entry is created in the packaging control file, pcd.psf, where TOSTAGE is /home/mybb/inst.images/hp9000_ux_10/shipdata:

    postinstall  /home/mybb/inst.images/hp9000_ux_10/shipdata/odehello.bin.post_i
    preremove    /home/mybb/inst.images/hp9000_ux_10/shipdata/odehello.bin.pre_d
    Similarly, when the mk parse_all command is executed on AIX, the configuration scripts are again assumed to reside in the TOSTAGE directory, so the following entry is created in the packaging control file, pcd.pd, where TOSTAGE is /home/mybb/inst.images/rios_aix_4/shipdata: 
    control_files = [ /home/mybb/inst.images/rios_aix_4/shipdata/odehello.bin.post_i 
                  /home/mybb/inst.images/rios_aix_4/shipdata/odehello.bin.pre_d 
                ]
    On RPM, the entry in the CMF for the above mentioned config files will look like

         configFiles = [ ("post"  "odehello.bin.post_i")
                         ("preun" "odehello.bin.pre_d") ]

    and the following entry is created in the packaging control file, pcd.spec

         %post bin
         ....... (Contents of odehello.bin.post_i)
         %preun bin

    SMP/E Packaging with ODE

    System Modification Program Extended (SMP/E) is a tool that manages the installation of software products on OS/390 (MVS) systems and tracks modifications made to these products. The package processed by SMP/E is composed of two distinct types of data: instructions to SMP/E identifying the software elements of the package and how to install them, and the software elements themselves. The instructions to SMP/E in a package are called modification control statements (MCS). ODE packaging provides mechanisms to create the necessary MCSs in a package so it can be installed using SMP/E.

    The following section provides the guidelines on how to use ODE for packaging using SMP/E.

    MVS Packaging with SMP/E in ODE

    ODE provides a way to specify MCS instructions for every part included in the package through the use of a Common Metadata File (CMF). The various types of stanzas and the valid attributes for each stanza that are allowed in the CMF are described in the section Common Metadata File.

    The CMF is parsed by ODE into a platform-specific control file. On USS, this control file is pcd.mvs which will be used by the gather and runpkgtool targets of ODE. The gather target of ODE will ensure the necessary files to be packaged are copied from the USS side to the TSO side. The runpkgtool target of ODE will generate JCL which calls RTG to create the package. The JCL that is generated contains all of the necessary MCS information, originally specified in the CMF, for RTG to properly create the package for SMP/E installation.

    Link-Edit Support in ODE

    Overview of the ODE link-edit process

    In the ODE CMF, load modules are defined to the CMF with a fileType of 'MOD', and are intended to refer to the pre-linked object modules used as input to the link-edit process. These pre-linked object modules must be link-edited into load modules, which are then shipped with the resultant package.  ODE facilitates the copying of these pre-linked object modules into a temporary dataset, and the generation and execution of the necessary jobsteps to perform the link-edit of each part into a load module suitable for shipping. The jobs that are eventually executed will invoke the IEWL linker with the NCAL option to link-edit the necessary parts on TSO without resolving external references. Once link-edited, these parts are now in their final shippable format.
    The modules in the CMF with fileType of 'PROGRAM' can also be link-edited. The same rules apply for these modules as they do for modules with fileType of 'MOD'.

    Define the distribution library

    Users should specify allocation parameters for the shippable RECFM U distribution library, used to hold the link-edited load modules, in the CMF via the distlibs attribute.  ODE will handle all processing relating to the allocation and population of the FB 80 temporary distribution library that will serve as input to the link-edit process.  Also, the targetDir attribute for any load module to be link-edited should refer to the RECFM U distribution library the load module will be shipped in, and not the temporary FB 80 distribution library.

    Specify link-edit parameters for ++MOD and ++PROGRAM parts

    An additional attribute must be defined in the CMF File stanza to denote that this part should be link-edited.  The attribute lkedParms provides a means for users to specify link-edit parameters for that particular load module.  Any 'MOD' or 'PROGRAM' part that contains the lkedParms attribute will be assumed to be a pre-linked object module that must be included in the link-edit step.  A 'MOD' or 'PROGRAM' part without lkedParms will not be included in the link-edit step. An optional attribute lkedCond can be used for each link-edited part to specify the maximum error code allowed before that part can be link-edited. This value is set to 4 by default for all the link-edited parts.

    ODE Actions

    The standard ODE packaging actions can be used to generate the link-edited load modules:

    parse_all

    • parse the CMF to obtain the necessary link-edit parameters
    gather_all
    • ODE will allocate the temporary FB 80 distribution library used as input to the link-edit process.  The size of the dataset is determined by adding the parameters of all RECFM U datasets allocated by the user via the distlibs attribute.
    • ODE will then populate the temporary FB 80 dataset with the pre-linked object modules marked for link-editing via the lkedParms attribute.
    • ODE will then generate and execute the job steps to perform the link-editing of the pre-linked object modules into their appropriate load modules.  These link-edit job steps are executed as part of a separate JCL job than the job run during gather_all to copy the parts to their respective distribution libraries.
    runpkgtool_all
    • the link-edited load modules are included as shippable parts in the resulting package.

    JCLIN Generation in ODE

    Overview of JCLIN

    All shippable files of type ++MOD may need to be link-edited at installation time by SMP/E into a load module.  In order to facilitate the creation of the load module, a JCLIN job must be executed against the necessary shippable parts.  This section describes how users can use the ODE CMF to automatically drive the creation of the JCLIN file and its inclusion in the resulting package.

    CMF Declarations

    To be included in the JCLIN generated by ODE, a part must be of type 'MOD' and have the lkedTo attribute specified.  The following list of CMF attributes are all used to generate the resulting JCLIN.  All attributes are optional and part of the File stanza unless otherwise noted.  See the preceding File Stanza documentation for more information.
    • lkedTo (required to mark a part for JCLIN generation)
    • lkedRc
    • hfsLkedName (required for JCLIN generation if lkedTo is 'HFS' or 'BOTH')
    • pdsLkedName (required for JCLIN generation if lkedTo is 'PDS' or 'BOTH')
    • hfsAlias
    • pdsAlias
    • setCode
    • jclinMode
    • entry
    • extAttr
    • jclinLkedParms (required for JCLIN generation)
    • include
    • sysLibs
    • order
    • jclinLib (MvsInfo stanza)
    • userId
    • groupId
    • permissions
    • targetFile (required)
    • targetDir or distLib (required)
    • partInfo
    • libraryDD
    • sysLibsLibraryDD

    ODE Actions

    The standard ODE packaging actions can be used to generate the JCLIN part:

    parse_all

    • ODE will generate the JCLIN file on HFS.
    • the JCLIN file will be created in the director PKG_CONTROL_DIR with a filename of <FMID>.jclin, where FMID is the name of the current FMID.
    • the JCLIN that is generated is only intended to run within the context of SMP/E, and is not designed to run outside of SMP/E.
    • can optionally define a file stanza in the CMF with hfsCopyType = 'jclin_only', denoting a file stanza solely used to generate JCLIN.  This is a virtual file that does not have to exist in the ship tree, and will not be packaged with the product.
    gather_all
    • ODE will copy the JCLIN to a distribution library on TSO defined by the jclinLib attribute.
    • If the distribution library specified by jclinLib is not contained in the distlibs attribute, ODE will automatically allocate the library using default parameters of LRECL=6160, Primary Blocks=5, Secondary Blocks=5.
    • Per the MVS Packaging Rules, the distribution library containing the JCLIN part must be FB80.
    • All files with hfsCopyType = 'jclin_only' will be ignored during this step.
    runpkgtool_all
    • A ++JCLIN statement will be written to the appropriate location in the ODE-generated DEFPKGS dataset, used as SMP input to the Build/390 Relfile Tape Generator (RTG) program.
    • If the sysLibs attribute is defined for any part in the JCLIN, the CALLLIBS keyword will be used with the ++JCLIN statement.
    • The only other supported keyword to the ++JCLIN statement is RELFILE(distlib), which will appear in every ++JCLIN statement generated by ODE.  distlib refers to the distribution library where the JCLIN part has been stored.
    • The resulting SMPMCS, generated by RTG, will refer to the JCLIN part as residing in the first RELFILE, per the MVS Packaging Rules.
    • All files with hfsCopyType = 'jclin_only' will be ignored during this step.

    InstallShield Usage with ODE

    The following section provides some guidelines on how to use ODE for packaging using InstallShield on Windows platforms.

    Creating a Project with InstallShield

    The following outlines how to create a project in InstallShield 5.5.  To use the packaging features of ODE on Windows, a project must first be created and built in InstallShield.  Consult an InstallShield manual for further information on how to create and manage projects in InstallShield.
    1. Initialize project - Create a project setup in InstallShield using either the Project Wizard, or an existing template.  Use of the Project Wizard will guide you through a series of steps that customize your installation sequence and produce a runnable setup that requires minimal modification.  For specific usage geared toward a specific type of project, a template may be used, which will pre-define many features of the installation.  For example, InstallShield 5.5 comes with built-in templates for MS Access, Visual Basic, and several other types of projects.
    2. Create File Groups - The next step in the project setup is to create file groups.  Files that are similar in function and type may be grouped together in an installation.  Such a group is known as a file group.  A file group may consist of several files and can refer to files with dynamic or static links (see File group file links for more detail).
    3. Create Components - Next, one or more components of the application must be defined.  A component is used to define a particular set of functionality in an application.  For instance an application that installs several products (for example, an editor, a compiler, a debugger), may make each product a separate component.  A component can contain specific file groups, and can also contain subcomponents, for further application refinement.
    4. Edit project script - If a project setup was created using either the Project Wizard or a pre-defined template, a project script already exists for the project.  This script does not need to be modified to handle the basics of executing the installation.  However, if additional functionality is desired, the script, written in InstallShield's "Install Script" language, may be edited.  The Function Wizard is available to assist in inserting a new function call with the appropriate parameters, or the script, "setup.rul", can be edited manually.  Again, this is an optional step and is not required to perform the basic steps of installation.
    5. Compile project script - After the install script has been created and/or edited, it must be compiled.  This can be done either through the InstallShield IDE, or by calling the command line tool compile.exe, as supplied with InstallShield.
    6. Create Media - Once the script is compiled, the next step is to create the media.  A 'media' consists of the final set of installable images that can be installed on a target system, and is specially formatted for different types of media (CD-ROM, floppy disk, etc.).  To build the media, the Media Build Wizard can be used, which will guide you through all of the options for creating and building a new install media.  This step must be done from the InstallShield IDE the first time a media is created for a project.  However, subsequent builds of the same media can be accomplished through the IDE or the command line tool isbuild.exe, as supplied with InstallShield.

    Creating a Single Installable Image with PackageForTheWeb

    A single installable image can be created for the project using PackageForTheWeb. This can be done by creating a new PackageForTheWeb project and choosing to create an Intel self-extracting executable. The installation steps need you to pick the source files needed for the package and the directories to create the project file (.pfw) and the installable image (.exe) file. The command line tool for executing this procedure is pftwwiz.exe as supplied by ProjectForTheWeb. Consult a PackageForTheWeb manual for more details on the usage.

    ODE Packaging Passes on Windows

    The following describes the three mk passes that pertain to packaging in ODE.
    • Gather - The gather pass of mk will copy shippable objects, once they have been built, into a location from which they can be used as input to the platform-specific packaging tools.  This does not differ from the gather pass on other platforms.
    • Parse - On most platforms, the parse pass of mk will parse the CMF (Common Metadata File) to obtain the installed images structure and properties, from which the platform-specific packaging input files will be generated.  However, on Windows, since the packaging input files must be created using InstallShield, and cannot be created outside of the InstallShield IDE, the parse action does not access the CMF, and simply outputs nothing.  Use of this pass is therefore not required on Windows.
    • Runpkgtool - On most platforms, the runpkgtool pass of mk will call the platform-specific packaging tools and, using the packaging input files from the parse pass and the shipped images from the gatherpass, generate installable images for the given platform.  However, on Windows, the installable images have to have been already created by the Media Build Wizard.  Therefore, the runpkgtool pass on Windows will simply update the current media set with new images, via a call to isbuild.exe, which will re-package the data with the latest set of files from the gather pass.  If the runpkgtool pass is run against a non-existent media, an error condition will result. This pass also creates a single installable image via a call to pftwwiz.exe by creating a projectfortheweb file.

    Other InstallShield Notes

    File group file links

    When creating a file group, individual files on the source machine are specified which compose a file group.  In versions of InstallShield prior to 5.5, these files had to be given a complete hard-coded path to their location on the source machine, meaning if the location of these files changed from one install to the next, the properties of the file group would also have to be changed to reflect the new location of the files.  However, in InstallShield 5.5, a new feature known as Dynamic File Links was implemented.  A "dynamic" file link is one composed of variables, subdirectories, and wildcards.  File links can now include:
    • Standard variables - These are variables defined within the InstallShield environment
    • Environment variables
    • Registry variables
    • Wildcards
    • An option to include all of the subfolders of a given location
    The use of environment variables can be especially useful in the ODE build environment.  Consider the example where it is desired to refresh the media data with new shipped images after a new build.  However, the base build directory changes from build to build (for example, b5, b6, b7, etc.).  In previous versions of InstallShield where the file links were hard-coded, the InstallShield project would have had to been manually updated to reflect the new location of the shipped images for each new build.  Now, in InstallShield 5.5 where dynamic file links are possible, the file links can be specified with an environment variable that reflects the location of the current build.  Consider the following file link designation:

            <TOSTAGE>\usr\odehello\html\*

    This uses the variable TOSTAGE, which in the ODE build environment, is defined as the base directory in which all built executable programs are installed, as performed by the gather pass.  Also, the use of the wildcard character '*' allows all files in the specified subdirectory to be installed, as opposed to having to manually specify each file.  Defining file links in file groups in this manner allows flexibility in a build environment to dynamically resolve the location of the files to be installed.
