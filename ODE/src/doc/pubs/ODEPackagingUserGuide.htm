About this document

This document describes the process involved for creating installable images using different packaging tools, on different platforms. This document assumes that the reader is familiar with ODE concepts like targets, passes, etc. It is recommended that the ODE Packaging Reference and ODE Packaging Tutorial be read before continuing with this document.

Setting up a sandbox/backing build for packaging

To start using the packaging features of ODE, there are several things that needs to be done like creating a CMF (Common Metadata File),  modifying the makefiles, rules to setup the parse_all, gather_all and runpkgtool_all targets. For more information on these targets and other packaging targets, please refer to the ODE Packaging Targets section of ODE Packaging Reference. ODE invokes different packaging tools on different platforms to create installable images. Please refer to the ODE Packaging Reference for the default packaging tools that ODE uses on different platforms.

Creating the CMF

The CMF is designed in such way that the metadata it contains can be mapped to different packaging tools specific metadata. The CMF contains information of each shippable file and information of package itself that is being created. Each packaging tool supports different package structures. These different packaging hierarchy structures can be represented in the CMF. Please refer to the ODE Packaging Reference for detailed description of the CMF.

A CMF can be created in two ways: by hand editing and by running mk cmfgen_all command. These two are described in the following two sections.

Hand editing

Even though, the CMF enables packaging tool independent way of expressing metadata, a CMF attribute might still need to have different values for different packaging tools. For example, entityId attribute of EntityInfo stanza represents the FMID when packaging with B/390 and it represents name of the package when packaging with pkgmk. The value of FMID needs to be in a specific format. Obviously, we cannot use the same value for the package then packaging with pkgmk. So, to accomplish this in an automated way, a C program with preprocessor statements can be created. These preprocessor statements will be based on the packaging tool being used to package.

InstallEntity
{
  EntityInfo
  {
    #ifdef pkgmk
      entityId = "odehello";
    #endif
    #ifdef b390
      entityId = "hbps112";
    #endif
  }
}

Running mk cmfgen_all command

This command executes a tool that takes makefile variables and install entity information file as input and generates the CMF. Please refer to the CMF Generation User's Guide to get detailed information on this way of generating the CMF. The install entity information file can still be generated using the previous method. For example, in a directory that contains one C file (foo.c), the makefile will look something like this:

# Should be already defined in makefiles
IDIR    = /usr/odehello/bin
ILIST   = foo.o
IMODE   = 755
IGROUP  = spatest
IOWNER  = staff
CMF_PARENT = odehello.bin

The resultant file stanza will be:

file
{
   fileType          = 'F';
   sourceFile        = "foo.o";
   sourceDir         = "/usr/odehello/bin/";
   targetDir         = "/usr/odehello/bin/";
   targetFile        = "foo.o";
   permissions       = "755";
   userId            = "spatest";
   groupId           = "staff";
 }

Setting up for parse_all

The parse_all target creates packaging tool specific control files using the information in the CMF. The parse_all target is executed by issuing mk parse_all command in a sandbox. mk parse_all does not do anything if the PARSE makefile variable is not defined. The parse_all target also requires the PKGFAMILY, PKG_CMF_FILE, PKG_CONTROL_DIR makefile variables to be defined. For example, define these makefile variables in the makefile of ${SANDBOXBASE}/src directory as follows:

PARSE=
PKGFAMILY=mkinstall
PKG_CMF_FILE=${SANDBOXBASE}/obj/${CONTEXT}/cmf/myCmf
PKG_CONTROL_DIR=${SANDBOXBASE}/inst.images/${CONTEXT}/madata

Doing so will enable to run mk parse_all in the src directory of a sandbox, which will create packaging tool specific control files in the directory specified by PKG_CONTROL_DIR using the CMF specified by PKG_CMF_FILE.

If there is more than one CMF that needs to be parsed in a sandbox then the CMFPARSE_SUBDIRS makefile variable needs to be defined in a makefile that instructs ODE to execute mk parse_all in the directories specified by the CMFPARSE_SUBDIRS makefile variable.

Setting up for gather_all

The gather_all target executes install_all target on all platforms and also runs installmvs_all target in addition to running install_all target on MVS/USS platform.

The install_all target gathers all the shippable files in one location. The install_all target recursively goes to the directories specified by the SUBDIRS makefile variable and moves the shippable files specified by the ILIST makefile variable to the ${TOSTAGE}/${IDIR} directory. For example, if there are some shippable files in the sub directories bin/client and bin/logger of src directory then define the SUBDIRS makefile variable in the makefile of ${SANDBOXBASE}/src directory as follows:

SUBDIRS=bin/client bin/logger

define TOSTAGE globally in the Makeconf as follows:

TOSTAGE=${SANDBOXBASE}/inst.images/${CONTEXT}/shipdata

define ILIST, IDIR in the makefile of bin/client directory as follows:

ILIST=client.o
IDIR=/usr/bin/client/

Similarly, define ILIST and IDIR in the makefile of bin/logger directory as follows:

ILIST=logger.o logger1.o
IDIR=/usr/bin/logger/

Doing so, if mk install_all is run in the src directory of a sandbox, will copy shippable files specified by ILIST (client.o, logger.o and logger1.o) from the OBJ tree to the ${TOSTAGE}/${IDIR} directory. So, client.o will be copied to ${SANDBOXBASE}/inst.images/${CONTEXT}/shipdata/usr/bin/client directory and logger.o, logger1.o will be copied to ${SANDBOXBASE}/inst.images/${CONTEXT}/shipdata/usr/bin/logger directory.

The installmvs_all target copies all the shippable files listed in the pcd.mvs (is created by the parse_all target) file, from the TOSTAGE area on the MVS/USS side of an MVS/OE system to a location on the TSO side. This target is valid only if the PKGFAMILY makefile variable is set to mvs. The installmvs_all target is executed by issuing mk installmvs_all command in a sandbox. mk installmvs_all does not do anything if the INSTALLMVS makefile variable is not defined. INSTALLMVS is already defined in the rules if the platform is MVS. The installmvs_all target requires the PKG_MVS_JOBCARD, PKG_MVS_USERID, PKG_MVS_PASSWORD, PKG_MVS_EXEC_DATASET makefile variables to be defined. For example, define following makefile variables globally in the Makeconf file:

PKG_MVS_JOBCARD=/home/aambati/jobcard
PKG_MVS_USERID=anil
PKG_MVS_PASSWORD=xxx
PKG_MVS_EXEC_DATASET=aambati.rexx.exec

Doing so will enables to run mk installmvs_all in the src directory, which will copy the shippable files from the TOSTAGE area in the sandbox to the datasets on TSO side. These dataset names are of form <HLQ>.<FUNCTION>.<DISTLIB>.

Setting up for runpkgtool_all

The runpkgtool_all target runs the packaging tool that is specified by the PKGFAMILY makefile variable to create an installable image. The PKGFAMILY is set to a default packaging tool for each platform. In order to use a different packaging tool on a particular platform, redefine the PKGFAMILY. For example, to use RPM on Solaris platform, define PKGFAMILY in the Makeconf file as follows:

PKGFAMILY=rpm

The runpkgtool_all target requires the PKG_OUTPUT_DIR, PKG_CONTROL_DIR and TOSTAGE makefile variables to be defined. There are other makefile variables that are required depending the packaging tool. Please refer to the ODE Packaging Reference for the list of these makefile variables. For example, define the following makefile variable in the makefile of ${SANDBOXBASE}/src directory:

RUNPKGTOOL=

The TOSTAGE and PKG_CONTROL_DIR are defined as mentioned in the above examples. The PKG_OUTPUT_DIR is set to ${PACKAGEBASE}${DIRSEP}images in the rules. When mk runpkgtool_all is run in the src directory of a sandbox, it will create a installable image in the directory specified by PKG_OUTPUT_DIR.

Packaging with different Tools

ODE invokes different packaging tools to create installable images on different platforms. These packaging tools support different packaging paradigms. Each tool require data to be represented in different ways.

RPM

RPM (Redhat Package Manager) is the default packaging tool on all the linux platforms. But, RPM can also be used on platforms like solaris and AIX. ODE will use the RPM build command to build a binary package for installation on a target machine. The CMF describes the installable image and the information in the CMF is used to generate a spec file that is used by RPM's build command. A spec file is the heart of RPM's packaging process; it could describe a package and/or one or more sub packages of a product. Please refer to Maximum RPM for more information on packages and sub packages.

Creating the CMF

Understanding how CMF's InstallEntity and File stanzas can be used to produce the spec file is the key to packaging with RPM. Each InstallEntity stanza could either describe a package or a sub package. The sub stanzas of an InstallEntity stanza can then be used to describe a package or a sub package. Provides
The Provides tag is used to specify a ``virtual package'' that the packaged software makes available when it is installed. Normally, this tag would be used when different packages provide equivalent services. For example, any package that allows a user to read mail might provide the mail-reader virtual package. The Requires tag is used to alert RPM to the fact that the package needs to have certain capabilities available in order to operate properly. These capabilities refer to the name of another package, or to a virtual package provided by one or more packages that use the provides tag. The Conflicts tag is the logical complement to the requires tag. The requires tag is used to specify what packages must be present in order for the current package to operate properly. The data is stored in the package however, and will be written into RPMs database when the package is installed.

The following two sections will describe how spec files that describe a product with sub packages and a product with no sub packages can be represented in the CMF.

One or more sub packages

The relationship between File stanzas and InstallEntity stanza is established with immChildFiles attribute of LinkInfo stanza. The relationship between two InstallEntity stanzas is established with immChildEntities of LinkInfo stanza. An InstallEntity stanza cannot have both immChildFiles and immChildEntities attributes at the same time. So, ODE does not support a spec file that describes a main package which has files in it and one or more sub packages.

In the example below, the main package does not contain any files but the sub package odebin contains files. The main package is represented by the first InstallEntity stanza. The sub package odebin is represented by the second InstallEntity stanza.

swpackage

swpackage is the default packaging tool on the HP-UX operating system. The CMF is used to create the Product Specification File (pcd.psf) that will be used by swpackage to create an installable image, which can be installed by the swinstall command. A PSF describes a package.

Creating the CMF

Determining the product structure is the key to the packaging with swpackage. A software package that a PSF describes should follow a predefined structure. This structure consists of products, subproducts, filesets and files. At the top of this hierarchical structure lies the product. A product is used to group filesets and/or subproducts. Optionally, subproducts are used to group related filesets with in a product if the product contains several filesets. Filesets are used to group the shippable files.

Even though, a PSF can define one or more products, a CMF can only describe one product. For more information on creating packages using swpackage, please refer to the Managing HP-UX software with SD-UX  guide.

A CMF InstallEntity stanza could describe either a product or a fileset. The sub stanzas of an InstallEntity stanza can then be used to describe a product or a class. The following table lists all the CMF attributes that are used to describe a product or a fileset and their mapping to the pcd.psf file entries.

Link-Edit Support in ODE

Overview of the ODE link-edit process
In the ODE CMF, load modules are defined to the CMF with a fileType of 'MOD', and are intended to refer to the pre-linked object modules used as input to the link-edit process. These pre-linked object modules must be link-edited into load modules, which are then shipped with the resultant package. ODE facilitates the copying of these pre-linked object modules into a temporary dataset, and the generation and execution of the necessary jobsteps to perform the link-edit of each part into a load module suitable for shipping. The jobs that are eventually executed will invoke the IEWL linker with the NCAL option to link-edit the necessary parts on TSO without resolving external references. Once link-edited, these parts are now in their final shippable format.

The modules in the CMF with fileType of 'PROGRAM' can also be link-edited. The same rules apply for these modules as they do for modules with fileType of 'MOD'.

Define the distribution library
Users should specify allocation parameters for the shippable RECFM U distribution library, used to hold the link-edited load modules, in the CMF via the distlibs attribute. ODE will handle all processing relating to the allocation and population of the FB 80 temporary distribution library that will serve as input to the link-edit process. Also, the targetDir/distLibattribute for any load module to be link-edited should refer to the RECFM U distribution library that the load module will be shipped in, and not the temporary FB 80 distribution library.
Specify link-edit parameters for ++MOD and ++PROGRAM parts
An additional attribute must be defined in the CMF File stanza to denote that this part should be link-edited. The attribute lkedParms provides a means for users to specify link-edit parameters for that particular load module. Any 'MOD' or 'PROGRAM' part that contains the lkedParms attribute will be assumed to be a pre-linked object module that must be included in the link-edit step. A 'MOD' or 'PROGRAM' part without lkedParms will not be included in the link-edit step.
ODE Targets
The standard ODE packaging targets can be used to generate the link-edited load modules:

parse_all

  • parse the CMF to obtain the necessary link-edit parameters
gather_all
  • ODE will allocate the temporary FB 80 distribution library used as input to the link-edit process. The size of the dataset is determined by adding the parameters of all RECFM U datasets allocated by the user via the distlibs attribute.
  • ODE will then populate the temporary FB 80 dataset with the pre-linked object modules marked for link-editing via the lkedParmsattribute.
  • ODE will then generate and execute the job steps to perform the link-editing of the pre-linked object modules into their appropriate load modules. These link-edit job steps are executed as part of a separate JCL job than the job run by the gather_all to copy the parts to their respective distribution libraries.
runpkgtool_all
  • the link-edited load modules are included as shippable parts in the resulting package.

JCLIN Generation in ODE

Overview of JCLIN
All shippable files of type ++MOD may need to be link-edited at installation time by SMP/E into a load module. In order to facilitate the creation of the load module, a JCLIN job must be executed against the necessary shippable parts. This section describes how users can use the ODE CMF to automatically drive the creation of the JCLIN file and its inclusion in the resulting package.
CMF Declarations
To be included in the JCLIN generated by ODE, a part must be of type 'MOD' and have the lkedTo attribute specified.  The following list of CMF attributes are all used to generate the resulting JCLIN.  All attributes are optional and part of the File stanza unless otherwise noted.
  • lkedTo (required to mark a part for JCLIN generation)
  • lkedRc
  • hfsLkedName (required for JCLIN generation if lkedTo is 'HFS' or 'BOTH')
  • pdsLkedName (required for JCLIN generation if lkedTo is 'PDS' or 'BOTH')
  • hfsAlias
  • pdsAlias
  • setCode
  • jclinMode
  • entry
  • extAttr
  • jclinLkedParms (required for JCLIN generation)
  • include
  • sysLibs
  • order
  • jclinLib (MvsInfo stanza)
  • userId
  • groupId
  • permissions
  • targetFile (required)
  • targetDir or distLib (required)
  • partInfo
  • libraryDD
  • sysLibsLibraryDD
ODE Actions
The standard ODE packaging commands can be used to generate the JCLIN part. Note: mk parse_all should be executed before mk gather_all for packaging with RTG when creating an IPP package.

parse_all

  • ODE will generate the JCLIN file on HFS.
  • the JCLIN file will be created in the directory PKG_CONTROL_DIR with a filename of <entityName>.jclin, where entityName is the name of the current Function.
  • the JCLIN that is generated is only intended to run within the context of SMP/E, and is not designed to run outside of SMP/E.
gather_all
  • ODE will copy the JCLIN to a distribution library on TSO defined by the jclinLib attribute.
  • If the distribution library specified by jclinLib is not contained in the distlibs attribute, ODE will automatically allocate the library using default parameters of LRECL=6160, Primary Blocks=5, Secondary Blocks=5.
  • Per the MVS Packaging Rules, the distribution library containing the JCLIN part must be FB 80.
runpkgtool_all
  • A ++JCLIN statement will be written to the appropriate location in the ODE-generated DEFPKGS dataset, used as SMP input to the Build/390 Relfile Tape Generator (RTG) program.
  • If the sysLibs attribute is defined for any part in the JCLIN, the CALLLIBS keyword will be used with the ++JCLIN statement.
  • The only other supported keyword to the ++JCLIN statement is RELFILE(distlib), which will appear in every ++JCLIN statement generated by ODE.  distlib refers to the distribution library where the JCLIN part has been stored.
  • The resulting SMPMCS, generated by RTG, will refer to the JCLIN part as residing in the first RELFILE, per the MVS Packaging Rules.

VPL Support in ODE

Overview
ODE faciliates the creation of VPLs (Virtual Private Listings). The TERSE or TRSMAIN programs are used to create the VPLs. The location of these programs should be specified to ODE by defining the PKG_MVS_TERSE_CLIST_DATASE and PKG_MVS_TERSE_LOAD_DATASET makefile variables. Each VPL file is created in a separate sequential dataset. The name of this sequential dataset is of form:

<applid>.<entityName>.<targetFile>.LISTING

A file listing all the VPL file dataset names is created in a sequential data set. The name of this sequential dataset is of form:

<applid>.<entityName>.VPL.FILES

CMF Declarations
Each shippable part with shipType 'vpl' will be used to create VPL file. The TERSE/TRSMAIN program takes a part of shipType 'vpl' and the attributes listed below as input and creates a VPL file. If there is atleast one shippable file of shipType 'vpl' then, ODE creates and submits a JCL job to create VPL files during the execution of runpkgtool_all target. The following table lists all the VPL related CMF attributes and the corresponding VPL header entry.
 
CMF Attribute Stanza Corresponding VPL header entry Is the CMF attribute required?

vplFromSys VplInfo FROMSYS Required
vplAckn VplInfo ACKN Optional. Default value is 'ERROR'

InstallShield

The following section provides some guidelines on how to use ODE for packaging using InstallShield on Windows platforms.

Creating a Project with InstallShield

The following outlines how to create a project in InstallShield 5.5. To use the packaging features of ODE on Windows, a project must first be created and built in InstallShield.  Consult an InstallShield manual for further information on how to create and manage projects in InstallShield.
  1. Initialize project - Create a project setup in InstallShield using either the Project Wizard, or an existing template. Use of the Project Wizard will guide you through a series of steps that customize your installation sequence and produce a runnable setup that requires minimal modification. For specific usage geared toward a specific type of project, a template may be used, which will pre-define many features of the installation. For example, InstallShield 5.5 comes with built-in templates for MS Access, Visual Basic, and several other types of projects.
  2. Create File Groups - The next step in the project setup is to create file groups. Files that are similar in function and type may be grouped together in an installation.  Such a group is known as a file group. A file group may consist of several files and can refer to files with dynamic or static links (see File group file links for more detail).
  3. Create Components - Next, one or more components of the application must be defined. A component is used to define a particular set of functionality in an application. For instance an application that installs several products (for example, an editor, a compiler, a debugger), may make each product a separate component. A component can contain specific file groups, and can also contain subcomponents, for further application refinement.
  4. Edit project script - If a project setup was created using either the Project Wizard or a pre-defined template, a project script already exists for the project.  This script does not need to be modified to handle the basics of executing the installation. However, if additional functionality is desired, the script, written in InstallShield's "Install Script" language, may be edited. The Function Wizard is available to assist in inserting a new function call with the appropriate parameters, or the script, "setup.rul", can be edited manually. Again, this is an optional step and is not required to perform the basic steps of installation.
  5. Compile project script - After the install script has been created and/or edited, it must be compiled. This can be done either through the InstallShield IDE, or by calling the command line tool compile.exe, as supplied with InstallShield.
  6. Create Media - Once the script is compiled, the next step is to create the media. A 'media' consists of the final set of installable images that can be installed on a target system, and is specially formatted for different types of media (CD-ROM, floppy disk, etc.). To build the media, the Media Build Wizard can be used, which will guide you through all of the options for creating and building a new install media. This step must be done from the InstallShield IDE the first time a media is created for a project. However, subsequent builds of the same media can be accomplished through the IDE or the command line tool isbuild.exe, as supplied with InstallShield.

Creating a Single Installable Image with PackageForTheWeb

A single installable image can be created for the project using PackageForTheWeb. This can be done by creating a new PackageForTheWeb project and choosing to create an Intel self-extracting executable. The installation steps need you to pick the source files needed for the package and the directories to create the project file (.pfw) and the installable image (.exe) file. The command line tool for executing this procedure is pftwwiz.exe as supplied by ProjectForTheWeb. Consult a PackageForTheWeb manual for more details on the usage.

ODE Packaging targets

The following describes the three mk targets that pertain to packaging in ODE.
  • gather_all : The gather_all target will copy shippable objects, once they have been built, into a location from which they can be used as input to the platform-specific packaging tools.
  • parse_all : For packaging with most of the packaging tools, the parse_all target will parse the CMF to create packaging tool specific control files. However, for packaging with InstallShield,  input files required for packaging cannot be created outside of the InstallShield IDE; so, the parse_all target does not do anything.
  • runpkgtool_all : Usually, the runpkgtool_all target will invoke the packaging tools and, using the packaging input files from the parse_all target and the shipped images from the gather_all target, generate installable images. However, for packaging with InstallShield, the installable images have to have been already created by the Media Build Wizard. Therefore, the runpkgtool_all target will simply update the current media set with new images, via a call to isbuild.exe, which will re-package the data with the latest set of files from the gather_all target. If the runpkgtool_all target is run against a non-existent media, an error condition will result. This target also creates a single installable image via a call to pftwwiz.exe by creating a projectfortheweb file.

Other InstallShield Notes

File group file links
When creating a file group, individual files on the source machine are specified which compose a file group. In versions of InstallShield prior to 5.5, these files had to be given a complete hard-coded path to their location on the source machine, meaning if the location of these files changed from one install to the next, the properties of the file group would also have to be changed to reflect the new location of the files. However, in InstallShield 5.5, a new feature known as Dynamic File Links was implemented. A "dynamic" file link is one composed of variables, subdirectories, and wildcards. File links can now include:
  • Standard variables - These are variables defined within the InstallShield environment
  • Environment variables
  • Registry variables
  • Wildcards
  • An option to include all of the subfolders of a given location
The use of environment variables can be especially useful in the ODE build environment. Consider the example where it is desired to refresh the media data with new shipped images after a new build. However, the base build directory changes from build to build (for example, b5, b6, b7, etc.).  In previous versions of InstallShield where the file links were hard-coded, the InstallShield project would have had to been manually updated to reflect the new location of the shipped images for each new build. Now, in InstallShield 5.5 where dynamic file links are possible, the file links can be specified with an environment variable that reflects the location of the current build.  Consider the following file link designation:

        <TOSTAGE>\usr\odehello\html\*

This uses the makefile variable TOSTAGE, which in the ODE build environment, is defined as the base directory in which all built executable programs are installed, as performed by the gather_all target.  Also, the use of the wildcard character '*' allows all files in the specified subdirectory to be installed, as opposed to having to manually specify each file.  Defining file links in file groups in this manner allows flexibility in a build environment to dynamically resolve the location of the files to be installed.

ISMP

ODE supports updating of an installable image created with ISMP (InstallShield Multi Platform). ISMP is a java based packaging tool that is capable of creating installable images on different platforms like, Windows, AIX, Linux, Solaris. Here are the steps involved to update an installable image using ODE:
  • Create an ISMP project using the GUI IDE and compile it.
  • Specify the location of the project file by defining the makefile variable PKG_ISMP_PROJFILE
  • Specify the directory where the ISMP is installed by defining the makefile variable PKG_ISMP_INSTALLDIR.
  • Optionally, define these makefile variables: PKG_ISMP_OTHERCLASSES, USE_PKG_ISMP_REL_CLASSPATH, PKG_ISMP_CLASSES
  • Create the directory structure for the shippable files, that is expected by the ISMP project by running the gather_all target.
  • Run mk runpkgtool_all command in a sandbox. Please follow the steps outlined in the Setting up a sandbox/backing build for packaging  section to setup the sandbox for running the gather_all and runpkgtool_all targets.
Please refer to the runpkgtool_all section of ODE Packaging Reference for detailed explanation of each of the above makefile variables.
